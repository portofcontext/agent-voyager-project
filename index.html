Agent Execution
Protocol

A structured channel for agent observability. Any runner, any model, any SDK. Where execution flows clearly and every turn is accounted for.

Read the spec Python SDK GitHub

Two interfaces. Nothing more.

AEP defines a single inbound configuration and a single outbound event stream between orchestrators and runners. The boundary stays clean regardless of what connects on either side.

Config

The orchestrator passes its configuration to the runner once at startup: model, prompt, system prompt, tools, boundary limits. The runner reads it, then begins. Delivered over stdio locally or HTTP remotely.

Event stream

The runner emits NDJSON as it flows: one event per line, flushed immediately. Every tool call, every model turn, every cost update. Identical schema over stdio or HTTP/SSE.

A run charted

Every AEP-compliant run produces a stream you can store, replay, and audit, independent of the framework that generated it.

Eight requirements

A runner that satisfies these anchors to the protocol, regardless of which SDK, model, or language it uses.

  • 1
    Read a valid AEP config JSON from the inbound channel before starting.
  • 2
    Emit agent_start as the first event, including prompt and tools when available.
  • 3
    Emit tool_call before each tool invocation and tool_result after.
  • 4
    Emit model_turn_start and model_turn_end around each model inference.
  • 5
    Emit cost_update at least once per turn or after each model call.
  • 6
    Emit agent_stop as the final event.
  • 7
    All output is valid NDJSON: one JSON object per line, no pretty-printing.
  • 8
    Flush after each line. Consumers must be able to follow the stream as it arrives.

Principles

Vendor-agnostic

AEP carries no dependency on any orchestrator, cloud provider, or model. It is a specification, not a library. Any system that speaks the protocol can be observed.

Transport-agnostic

The same schema flows over stdio locally or HTTP/SSE remotely. No daemons, no sidecars, no observer dependencies on the consumer side.

Structured, not closed

Custom event types are valid and pass validation. Namespace them with a prefix to avoid drift against future AEP events. The structure guides without constraining.

Built to verify

The Python SDK ships a compliance validator. Pass it any NDJSON stream and receive a structured report of violations. The protocol makes its expectations explicit.
